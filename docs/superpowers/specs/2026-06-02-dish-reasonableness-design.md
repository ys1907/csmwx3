# 设计：菜品合理性治理（全库逐道定位）

- 日期：2026-06-02
- 状态：设计已确认（待用户复审本文档）
- 范围：全库 479 道逐道核准"定位" + 新增约 60 道家常菜 + 管理页"隐藏已停用"开关 + 版本/测试配套。以数据治理为主，少量页面改动。
- 关联：延续 `2026-06-02-data-fixes-design.md`。本次发现：上次"291 条旧分类落盘归一"因**沙箱隔离 node 写入实际未落盘**，仅靠运行时 `migrateFood` 兜底——本次一并真正落盘。

## 背景

用户反馈：盲盒抽到的菜"太异常、不像中国人今天会吃的"。核准后归纳为四个病灶（均需治理）：

1. **异国太杂**——首页池 29 道异国，含印尼炒饭 / 椰浆饭 / 炒粿条等小众项。
2. **地方太小众**——羊肉泡馍 / 豆花饭 / 裤带面 / 油泼扯面 / 米皮等出了产地少见。
3. **内脏猎奇**——猪脏粉 / 肥肠粉等劝退一部分人。
4. **格式单调**——首页池 110 道里 98 道是盖饭 / 面（89%），几乎没有家常炒菜、火锅、烧烤、早餐、汤；而这些**库里都有，只是 `defaultPoolWeight` 被设为 0 雪藏**。

根因洞察：库里**不缺**"中国日常菜"，缺的是**正确的定位**（权重 / 分类 / 场景 / 成餐语义）。因此本次目标不是"删菜"，而是**给每道菜一个最准确的定位**。

核心约束（沿用项目铁律）：原生微信小程序，无后端、无第三方依赖；纯逻辑与 `wx` 隔离、可单测；**本 agent 沙箱隔离 node 对工作区的写入**，故数据落盘走 Edit/Write 工具或用户本地脚本。

## 目标与非目标

**目标**
- 全库 479 道逐道核准"定位"（见第一节）；首页盲盒（及全库）出菜符合中国人日常。
- 新增约 60 道高频家常菜，补齐"在家做饭"维度。
- 管理页可隐藏已停用菜，兑现"全库干净"且不丢数据。

**非目标（避免膨胀）**
- 不改任何现有 `emoji` / 菜名（仅新增菜需要起名）。
- 不重写盲盒 / 凑一桌核心逻辑；纯炒菜沿用现有 `buildMealCombo` 承接。
- **不硬删任何记录**——不需要的菜一律 `enabled=false` + 权重 0，由管理页开关折叠（用户已选非破坏性方案）。
- 不引入任何新依赖或后端。

---

## 一、"定位"定义（每道菜核准的 6 维 + 处置）

遍历全库时，每道菜核准下列维度：

| 维度 | 取值 | 说明 |
|---|---|---|
| `category` | 新 12 分类之一 | 取最准的；**落盘纠正** 291 条残留旧分类 + 任何错分 |
| `defaultPoolWeight` | 0 ~ 1.0 | 按"日常频率 × 能否当主角"定档；0 = 出池 |
| `enabled` | true / false | 内脏猎奇 / 出库项置 false |
| `scene` / `scenes` | 外卖 / 堂食 / 自己做 / 食堂 | 实际食用场景 |
| `canBeMeal` / `mealPeriods` | bool / 时段数组 | 能否单独成一餐、属哪个时段 |
| 处置 | keep / 激活 / 出池 / 调权重 / 新增 | 该菜最终动作 |

**只改字段值、不改字段 shape**，故 `migrateFood` 不变、现有 65 个测试不受影响。

**`category` 保持单值（不做多分类）**：跨维度的"心情"（辣 / 清淡 / 早餐 / 奢侈…）由后续『盲盒心情 chip』以 per-field 谓词实现——辣·清淡 = `tags`、早餐 = `mealPeriods`、奢侈 = `budget`、家常菜 = `category`。一道菜可同时命中多个 chip（如麻婆豆腐 = 辣 + 家常菜），无需多分类、零 schema 改动。横跨两类的菜（酸菜鱼 / 大盘鸡）沿用"拆成两条记录"（家常版 + 火锅 / 面版）。

## 二、判断标准（核准定位的规则）

### 异国菜（cuisine = 日韩 / 东南亚 / 西式）
- **留**（国民级）：寿司、各式拉面、咖喱饭、韩式拌饭、石锅拌饭、泡菜炒饭、照烧鸡饭、乌冬、冷面、汉堡（各款）、披萨、海南鸡饭
- **出池**：东南亚小众（印尼炒饭 / 椰浆饭 / 炒粿条 / 咖喱牛腩饭）、偏正餐 / 猎奇日料（天妇罗盖饭 / 鳗鱼饭 / 炸虾盖饭 / 三文鱼饭）
- 硬约束：池内异国占比 ≤ ~12–15%
- 一句话判据：**一二线城市外卖 / 商场随处可点 = 留；要专门找特定店才有 = 出池**

### 地方小众
- **出池**：羊肉泡馍、豆花饭、裤带面、宜宾燃面、油泼扯面、米皮、擀面皮
- **留**：已全国普及的地方名吃（兰州拉面、重庆小面、武汉热干面、螺蛳粉、过桥米线、河南烩面、煲仔饭、黄焖鸡米饭…）

### 内脏 / 猎奇
- **全部出池**（`enabled=false` + 权重 0）：猪脏粉、肥肠粉、生鱼片、醉蟹、美蛙鱼头等
- 例外：凤爪等国民级小吃不算猎奇，保留

### 格式单调（"激活 / 补"，非删）
- 激活库里被雪藏的完整餐进池：火锅（库 15 道）、烧烤（库 15 道）、早餐（粥 / 包子 / 肠粉 / 胡辣汤 / 煎饼…）、经典套餐
- 新增约 60 道家常菜（见第四节）

## 三、权重体系（频率 × 主角度）

| 类型 | 权重 | 含义 |
|---|---|---|
| 家常盖饭 / 经典面 / 中式快餐 | 1.0 | 天天可能吃 |
| 硬菜·大荤（红烧肉 / 酸菜鱼 / 大盘鸡 / 糖醋排骨 / 黄焖鸡） | 0.8–1.0 | 能当主角 |
| 家常荤·半荤（西红柿炒鸡蛋 / 青椒炒肉 / 宫保鸡丁） | 0.6–0.8 | 日常主力 |
| 异国国民级 / 地方变体 | 0.45–0.65 | 调味，不喧宾夺主 |
| 火锅 / 烧烤 | 0.35–0.45 | 偶尔的共享大餐，出现但不刷屏 |
| 早餐 | 0.6–0.8 | 仅早餐时段内（`mealPeriods`） |
| 纯素 · 凉菜 | 0.15–0.3 | 极少单独出现 |
| 汤 | ~0.2 | 单独少出，主要进凑一桌 |

**设计意图**：素 / 凉 / 汤单独出现概率压到很低，但在「凑一桌」（一荤一素一汤）里担纲——既不让 `凉拌黄瓜` 单独刷屏，又不浪费。

## 四、家常菜整合

- **来源**：用户提供的 100 道高频家常菜。去重匹配结果：**43 道匹配到库内记录**（多为权重 0，激活即可；其中约 6 道"仅有盖饭版"、缺标准家常版，需另建）+ **57 道无匹配**。综合 → 实际**新增约 60 道、激活约 37 道**。
- **分类**：用现有 `家常菜`（炒 / 烧菜）、`汤粥炖品`（汤）、`配菜`（凉菜）；不新建分类。
- **场景**：与「自己做」强关联；`自己做` → 家常菜 + 凑一桌为主，`外卖 / 堂食` → 盖饭 / 快餐 / 火锅为主。
- **素 / 凉 / 汤**：低权重（见第三节）+ 凑一桌承接。
- **新增记录字段规范**：照既有条目补全全部字段；`emoji` 单字；`tags` 非空数组；新 `_id`（16 位十六进制）；`category` 用新 12 分类；`canBeMeal` / `mealPeriods` / `defaultPoolWeight` / `scene` / `scenes` / `availability` 据实填。

## 五、处理机制

- **出池 / 停用** = `defaultPoolWeight=0` + `enabled=false`（盲盒永不出）。
- **管理页"隐藏已停用"开关**：`enabled===false` 的菜默认折叠，顶部开关可展开查看。**移至后续 UI spec（§九）**——与心情 chip 同属 manage/index UI，一并做。本次数据治理只负责把不要的菜 `enabled=false`（盲盒已永不出）；它们暂时仍显示在管理页，UI spec 再加折叠开关。
- **落盘归一**：把磁盘上 291 条旧分类按权威映射真正写入 `data/foods.js`（用 Edit/Write 或用户本地脚本，不靠沙箱 node）。映射权威源：`scripts/cleanFoods.js` 及 `utils/util.js` 的 `LEGACY_CATEGORY_MAP` / `normalizeCategory`。
- **版本（只重置菜，不动其他）**：新增 `FOODS_SEED_VERSION='v4'`，**仅**用于 foods 重播种闸门——`index.js:167,206` / `manage.js:74` 的 `localVersion === APP_VERSION` 改为 `=== FOODS_SEED_VERSION`，`manage.js:150` 持久化同改。`APP_VERSION` 保持 `v3` 不动 → `history` / 收藏 / SSR 图鉴 / PK 全部保留，只有 foods 重播种（仅重置本地自定义菜，已确认接受）。⚠️ **绝不可直接 bump `APP_VERSION`**：storage key 全带它后缀，会连图鉴/历史一起清空。

## 六、全库遍历方法（实施时执行）

1. **分批**：按 `category` 切批（每批约 30–40 道），逐道按"第二节标准 + 第三节权重"核准 6 维定位，产出结构化定位表（`name` / `_id` → 新字段值 + 处置）。
2. **规模与准确性**：479 道判断量大 → 可用并行子代理分批审、主体汇总；每批结构化输出便于复核与抽查。
3. **落地（推荐用户本地跑脚本）**：定位表 → 补丁脚本，由用户在本地（无沙箱）执行——本地脚本能读到真实磁盘的 291 条旧分类并正确归一，`npm test` 也是真实结果；沙箱内 Edit 虽写得进真实文件，但 `node` 读到的是影子副本、验证不可信。新增约 60 条记录同批写入。

## 七、测试与验证

**测试**
- 不改字段 shape → `migrateFood` 不变、现有 **65 个测试保持绿**。
- 新增**数据体检测试**（`utils/foods.integrity.test.js`，直接 require `data/foods.js`）断言库级不变量：
  - `data/foods.js` 无残留旧分类（落盘归一验收）
  - 每条 `category` ∈ 新 12 分类；每条 `defaultPoolWeight` ∈ [0, 1]
  - 池内（`weight>0` 且 `enabled!==false`）无内脏猎奇关键词
  - 池内异国（cuisine 非中式）占比 ≤ 15%
  - 池内**非成餐项**（`canBeMeal===false`）权重 ≤ 0.3（确保素 / 凉 / 汤靠组合而非单独刷屏）
  - 池内项数 ≥ 80（防误删 / 空池）

**验证**
- `npm test` 全绿（**须在用户本地跑**）。
- ⚠️ **沙箱陷阱**：沙箱内 `node` / `npm test` 读的是 `data/foods.js` 的影子副本（category 已归一、与真实磁盘 291 条旧分类不符）→「无残留旧分类」等数据断言在沙箱内会**假性通过**。数据改动的权威验证一律在**用户本地（无沙箱）**进行，或用 Grep/Read 核对真实文件。
- **心情 chip 候选充足**（为后续盲盒 chip 特性铺路）：治理后「早餐 / 辣 / 清淡 / 家常菜 / 奢侈一顿」每个 chip 的**池内候选 ≥ ~15**。当前 3/5 近乎为空（奢侈 3 · 清淡 4 · 家常菜 2），靠激活 + 补菜 + 打标补足；并确保 `tags`(辣 / 清淡) · `budget`(💰💰💰) · `mealPeriods`(早餐) · `category`(家常菜) 准确填充。
- **抽样脚本** `scripts/samplePool.js`（模拟抽取、打印分布）：模拟 N 次盲盒抽取，输出格式 / 品类 / 异国占比分布，用数据证明"火锅烧烤早餐出得来、怪菜出不来"。
- 微信开发者工具实测：反复点盒子，确认"像中国人今天会吃的"。

## 八、风险与权衡

- **工作量**：新增约 60 条完整记录 + 全库逐道核准是主要成本。缓解：分批 + 子代理 + 结构化输出。
- **判断主观**：什么算"国民级 / 小众"有灰度。缓解：成文标准 + 抽样脚本 + 用户抽查兜底；灰度项倾向"降权保留"而非误删。
- **reseed 仅重置 foods**：经 `FOODS_SEED_VERSION` 隔离，只重置本地自定义菜；`history` / 收藏 / SSR 图鉴 / PK 不受影响（已确认）。
- **沙箱写入**：数据落盘统一走 Edit/Write 或用户本地脚本，避免 node 写入被隔离。
- **「奢侈一顿」chip 最薄**：全库仅 31 道 💰💰💰。治理时给火锅 / 烧烤 / 海鲜 / 西式硬菜标 💰💰💰 扩充到 ~20+；不足则把该 chip 放宽到含部分 💰💰。
